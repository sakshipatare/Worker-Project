# AI-Powered Worker Productivity Dashboard

This repository contains the backend and frontend for an AI-Powered worker productivity dashboard. It ingests computer-vision events generated by factory CCTV cameras, stores them in MongoDB, and displays productivity metrics.

## Architecture Overview

**Edge → Backend → Dashboard Architecture:**
1. **Edge (CCTV + CV Model)**: AI models deployed on edge devices analyze video frames. Upon detecting actions, they emit JSON events.
2. **Backend (Node.js/Express)**: Acts as an ingestion API layer. It validates and persists the JSON payloads into a MongoDB database. A separate endpoint computes metrics on-demand via data aggregations.
3. **Dashboard (React.js/Vite)**: Built a responsive dashboard with React and CSS, focusing on clarity of metrics and usability.

## Run Locally with Docker
Prerequisites: Docker and Docker Compose installed.

1. `git clone` the repository (or navigate to the workspace).
2. Run `docker-compose up --build`
3. Navigate to `http://localhost:8080` to see the dashboard.
4. Click **"Refresh Mock Data"** to automatically insert mock workers, workstations, and a simulated set of events for testing.

## Deployment to Render (Blueprint)

This project is configured for one-click deployment using **Render Blueprints**.

### 1. Database Setup (MongoDB Atlas)
Since Render does not provide a built-in MongoDB service, you must use an external provider like MongoDB Atlas:
1.  Sign up at [MongoDB Atlas](https://www.mongodb.com/cloud/atlas).
2.  Create a new **Free Shared Cluster**.
3.  In **Network Access**, allow access from `0.0.0.0/0` (or Render's IP range if preferred).
4.  In **Database Access**, create a user with a password.
5.  Click **Connect** -> **Drivers** -> Copy the **Connection String** (it looks like `mongodb+srv://...`).

### 2. Connect to Render
1.  Push your code to a GitHub repository.
2.  Go to the [Render Dashboard](https://dashboard.render.com/).
3.  Click **New +** -> **Blueprint**.
4.  Connect your GitHub repository.
5.  Render will detect the `render.yaml` file.
6.  Enter the `MONGODB_URI` (your MongoDB Atlas connection string) when prompted.
7.  Click **Apply**.

Render will automatically provision:
-   **factory-ai-backend**: A Docker-based Web Service.
-   **factory-ai-frontend**: A Static Site (optimized for Vite).

The frontend is automatically networked to communicate with the backend via the `VITE_API_URL` environment variable defined in the blueprint.


## Database Schema (MongoDB)
- **Worker**: `{ worker_id: String (Unique), name: String }`
- **Workstation**: `{ station_id: String (Unique), name: String }`
- **Event**: `{ timestamp: Date (Indexed), worker_id: String (Indexed), workstation_id: String (Indexed), event_type: String, confidence: Number, count: Number }`

## Metric Definitions
- **Active Time (Working Time)**: The time difference between a 'working' event and the next state-change event.
- **Idle Time**: Duration where the worker's latest state is "idle".
- **Utilization %**: `(Active Hrs / (Active Hrs + Idle Hrs)) * 100` (We intentionally exclude 'absent' from the denominator footprint unless expected factory-floor-time tracking requires its inclusion).
- **Units Per Hour (UPH)**: `Total Units Produced / Active Time in hours`.
- **Occupancy Hours (Station)**: Duration where ANY worker's state at the station is "working" or "idle".
- **Throughput Rate (Station)**: `Total Units Produced at Station / Occupancy Hours`.

## Assumptions & Tradeoffs
1. **State Sequences**: Assumed events like `working`, `idle`, and `absent` are "State Change" triggers. A state persists from $t_1$ to $t_2$. `product_count` triggers are point-in-time and don't interrupt ongoing time-durations.
2. **Time Boundaries**: Metrics are primarily calculated using event-bounded timestamps dynamically based on the available database data (min/max timestamps). The concept of an "8-hour shift" is only used in the API's seeded dummy data for simulation purposes.
3. **Aggregations in JS vs MongoDB (Performance)**: We currently compute metrics on-demand upon API request. This approach provides rapid logic iteration. However, for true scalability, metrics should be pre-aggregated using background jobs or stored in specialized summary collections.

## Theoretical Technical Solutions

### Handling Data Integrity Operations

**1. Intermittent Connectivity**
- **Edge Solution**: Implement a robust local caching mechanism on the edge devices (e.g., a lightweight local SQLite database). If the POST request fails to reach the backend, the edge device queues the event and retries with exponential backoff.

**2. Duplicate Events**
- **Backend Solution**: Event payloads should include a unique `event_id` or `trace_id` generated at the edge. The MongoDB schema would place a `unique: true` index on this ID, naturally dropping identical re-transmissions.

**3. Out-Of-Order Timestamps**
- **Backend Solution**: By sorting event queries securely by `{ timestamp: 1 }` (which works perfectly thanks to indexing) rather than relying on insertion payload order, metric calculators rebuild the exact contiguous chronological history regardless of variable transit delay.

### Model Lifecycles

**How would you add model versioning?**
- Add a `model_version: "v1.2.0"` field directly to every ingested metric event JSON. We can trace anomalous output metrics directly to the exact model code that generated them. 

**How would you detect model drift?**
- Create baseline "healthy" metrics (e.g. historical average `confidence` score per hour). If the rolling average `confidence` metric falls under a set threshold (e.g., drops from 0.95 to 0.75 for 3 consecutive hours), an alert fires.
- A secondary strategy would be to monitor real output distributions in BI tools and see if average worker utilization drastically spikes arbitrarily out of known human capability.

**How would you trigger retraining?**
- Build an automated data ingestion pipeline where edge devices flag low-confidence frames (e.g. `confidence < 0.6`).
- Upload those raw frames to a cloud bucket like AWS S3. 
- Trigger an automated script or scheduled background job that routinely pulls those low-confidence images, sends them to human-in-the-loop annotation tools, and subsequently fine-tunes the computer vision weights, publishing a new `model_version` OTA.

### Expanding Scalability
**How this scales from 5 cameras → 100+ cameras → multi-site:**
- **Ingestion Decoupling**: Introduce an asynchronous message broker (like Apache Kafka or RabbitMQ) between the Edge and the Backend. Events stream into Kafka rather than being synchronously awaited by an Express backend. 
- **Time-Series Database**: Pivot from traditional MongoDB to a dedicated Time-Series database (like InfluxDB or TimescaleDB) to handle millions of timestamped reads/writes.
- **Microservices**: Break the monolithic backend into dedicated services: `Ingestion API`, `Aggregation Engine`, and `Dashboard Serving API`, individually scaling horizontally via Kubernetes clusters across multiple geographic regions depending on the site.
